## 简介
服务层（`src/services`）提供可被背景服务调用的能力单元：AI 客户端初始化、URL 分类、扩展配置读写，以及两个基于 IndexedDB（`idb` 库）的持久化管理器。当前实际接入主流程的是 AI、分类服务、配置存储与 URL 分类数据库；事件数据库虽已实现但尚未被任何模块引用。

## AI 客户端 AI.ts
[src/services/AI.ts](file://src/services/AI.ts) 导出一个可变的模块级 `client`（`OpenAI | null`）与初始化函数 `initClient`：

```ts
export let client: OpenAI | null = null;

const baseUrls = {
  openai: "https://api.openai.com/v1",
  deepseek: "https://api.deepseek.com/v1",
};

export function initClient(provider: string, apiKey: string): OpenAI {
  client = new OpenAI({ baseURL: baseUrls[provider] ?? provider, apiKey });
  return client;
}
```

当 `provider` 是内置枚举时使用对应 base URL；否则 `provider` 本身被当作自定义 base URL。`baseURL` 与 `apiKey` 由 `Option` 提供。对应的 `host_permissions` 已在 `manifest.ts` 声明（`api.openai.com` / `api.deepseek.com`）。

## 分类服务 CategoryService
[src/services/CategoryService.ts](file://src/services/CategoryService.ts) 是 URL 分类的编排者，核心是 `getCategory(url, html, option?)`：

1. `extractDomain(url)` 提取域名；
2. `getCategoryFromDB` 先查 `urlCategoryDB.lookup`（命中缓存则直接返回）；
3. 未命中则 `loadOption` 读取配置、`initClient` 初始化 AI，再 `categorifyModel` 调用大模型；
4. 校验模型返回的类别属于 11 个合法 `UrlCategory` 之一；
5. 通过 `urlCategoryDB.put` 写回缓存。

`categorifyPrompt` 内置分类规则，要求模型输出两行文本（第一行 domain，第二行 category）。分类枚举来自 [src/models/types.ts](file://src/models/types.ts) 的 `UrlCategory`（共 11 类）。

## 配置存储 OptionStore
[src/services/OptionStore.ts](file://src/services/OptionStore.ts) 封装对 `chrome.storage.local` 的读写，存储键为 `"brainrest_option"`。因为 MV3 后台没有 `localStorage`，popup 与 service worker 统一走此 API。

- `loadOption()`：防御式读取并逐字段校验，任何缺失/非法都回退到 `DEFAULT_OPTION`（`openai` / `gpt-4o-mini` / 空 apiKey）。
- `saveOption()` / `clearOption()`：写入与清除，异常被吞掉，不阻断调用方。
- `normalizeProvider()`：仅接受 `openai`、`deepseek` 或以 `http(s)://` 开头的绝对 URL，其余回退默认。

配置结构定义于 [src/models/Option.ts](file://src/models/Option.ts)：`aiProvider`、`categorifyModel`、`apiKey`。

## URL 分类数据库 UrlCategoryDataBaseManager
[src/services/UrlCategoryDataBaseManager.ts](file://src/services/UrlCategoryDataBaseManager.ts) 是单例 `urlCategoryDB`，管理数据库 `brainrest_url_categories`（v1）中的 `url_categories` 对象仓库，主键为 `domain`，并建立 `domain`（唯一）、`category`、`updatedAt` 三个索引。

关键方法：

| 方法 | 说明 |
|------|------|
| `put` / `batchPut` | 写入/批量写入 `(domain → category)`，`updatedAt` 取当前时间 |
| `getExact` | 精确读取某 domain（不回溯） |
| `listByCategory` | 按类别反查所有 domain |
| `lookup` | 从二级域名逐级向上匹配到 5 级，命中即返回 |
| `lookupCategory` | `lookup` 的便捷封装，仅返回类别字符串 |
| `delete` / `clear` | 删除单条 / 清空 |

`lookup` 用 `MIN_LEVEL = 2`、`MAX_LEVEL = 5` 控制回溯层级，`normalizeDomain` 统一小写、去空白与尾部点。它既是分类结果缓存，也被 `SwitchEntropyAnalyzer` 用于跨类别转移熵计算。

## 事件数据库 EventDataBaseManager
[src/services/EventDataBaseManager.ts](file://src/services/EventDataBaseManager.ts) 是单例 `eventDB`，管理数据库 `brainrest`（v1）的 `events` 仓库，主键为 `timestamp`，含 `processed` 索引，并以 `DAY_MS`（24 小时）为界清理旧数据。提供 `batchPut`、`getUnprocessed`、`updateByTimestamp`、`markProcessed`、`prune` 等方法。

> [!NOTE]
> 该管理器已完整实现，但当前代码库中**从未被任何模块导入或调用**。疲劳引擎直接消费内存中的 5 秒滑动窗口队列，并不落盘事件。文档保留其说明以反映真实源码，但不应将其描述为主链路的一部分。
