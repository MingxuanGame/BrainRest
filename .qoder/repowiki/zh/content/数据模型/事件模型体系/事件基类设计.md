## 简介
`Event` 是所有事件模型的基类，定义了每一条采集事件都必须携带的公共字段，并配套一个工厂函数 `createEvent<T>` 负责填充公共字段的默认值。

## Event 接口
[src/models/events/Event.ts](file://src/models/events/Event.ts) 定义基类：

```ts
export interface Event {
  timestamp: number;   // 事件产生时间 (ms)
  type: string;        // 事件类型标识
  processed: 0 | 1;    // 是否已处理（0 未处理 / 1 已处理）
  url: string;         // 事件发生时的页面 URL
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | `number` | 由 `Date.now()` 填充 |
| `type` | `string` | 具体子类型以字面量收窄（如 `'keydown'`） |
| `processed` | `0 \| 1` | 处理标记，创建时为 `0` |
| `url` | `string` | 页面 URL；窗口事件为空串 |

## createEvent 工厂函数
`createEvent<T extends Event>` 接受某具体事件除公共默认字段外的属性，自动补齐 `processed: 0` 与 `timestamp: Date.now()`，返回完整事件对象。内容脚本与浏览器监听器均通过它构造事件，保证公共字段一致：

```ts
const event = createEvent<UiClickEvent>({
  type: "click",
  url: window.location.href,
  button, clientX, clientY, /* ... */
});
// 自动补齐 timestamp 与 processed
```

## processed 字段的作用
`processed` 沿用自"事件可落盘并标记处理状态"的设计（`EventDataBaseManager` 以 `processed` 建立索引并提供 `markProcessed`）。但当前主链路中，疲劳引擎直接消费内存 5 秒窗口队列，并未将事件写入 IndexedDB，因此 `processed` 在运行时始终为创建时的 `0`。此处如实说明，避免高估其在当前实现中的作用。
